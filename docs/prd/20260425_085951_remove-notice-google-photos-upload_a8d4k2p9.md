# PRD: 공지사항 Google Photos 업로드 제거

## 구체화 결과

| 단계 | 정리 |
|------|------|
| 존재 이유 | 공지사항 파싱 시 Google Photos 업로드가 더 이상 필요하지 않으며, 외부 API 토큰 의존성과 실패 지점을 줄인다. |
| 사용자 | 배치를 운영하는 관리자. 공지 요약 알림은 계속 받아야 하지만 Google Photos 앨범 적재는 원하지 않는다. |
| 핵심 장면 | 새 공지 감지 → 이미지 다운로드 → 이미지 분석/요약 → 카카오 공지 요약 전송 및 DB 기록. 이 흐름에서 Google Photos 업로드만 빠지고, 실패 시 재시도 후 `FAIL` 이력을 남긴다. |
| 경계 | 사진 게시판 카카오 이미지 전송, 카카오 이미지 업로드, 기존 Google Photos 클라이언트 파일 자체 삭제는 범위 밖이다. |
| 성공 기준 | 공지 이미지가 있어도 Google Photos API 호출 없이 요약이 전송되고, `config/google_token.json` 부재가 배치 시작 실패를 일으키지 않는다. 공지 처리가 최종 실패하면 어드민에서 `FAIL` 이력으로 확인 가능하다. |

## 프로젝트 개요

`naver-cafe-monitor`는 네이버 카페 게시판의 새 게시물을 감지해 사진/공지 알림을 카카오톡으로 전달하고 처리 이력을 저장하는 배치 및 관리 도구다. 이번 변경의 대상 사용자는 운영자이며, 목표는 공지사항 처리 경로의 불필요한 Google Photos 업로드를 제거해 운영 의존성을 낮추는 것이다.

## 톤·정체성

사용자 노출 문구 변경은 없다. 로그 문구는 기존 한국어 운영 로그 톤을 유지한다. 금칙어·브랜드 카피 변경은 해당 없음.

## 기능 요구사항

| ID | 요구사항 | 우선순위 | AC |
|----|----------|----------|----|
| F-01 | 공지사항 처리에서 Google Photos 업로드 토큰 생성과 앨범 추가 호출을 제거한다. | High | 공지 이미지 처리 테스트에서 `upload_images`/`add_to_album` 호출 경로가 없다. |
| F-02 | 배치 시작 시 Google Photos 클라이언트와 `config/google_token.json`이 필요하지 않게 한다. | High | `run()` 또는 공지 처리 함수 시그니처에서 Google Photos 의존성이 제거된다. |
| F-03 | 공지 이미지 다운로드, 이미지 분석, 카카오 요약 전송, DB 기록 동작은 유지한다. | High | 이미지 있는 공지 단위 테스트에서 `analyze_image`, `send_notice_summary`, `repo.save`가 호출된다. |
| F-04 | 사진 게시판과 카카오 이미지 전송 경로는 변경하지 않는다. | Mid | 관련 테스트 회귀가 통과한다. |
| F-05 | 공지사항 전용 Google Photos 앨범 ID/URL 상수를 제거한다. | Mid | `batch/src/batch.py`에 공지 Google Photos 앨범 상수가 남지 않는다. |
| F-06 | 공지 개별 게시글 처리 실패 시 최초 1회 시도 후 최대 3회 재시도한다. | High | 첫 분석 실패 후 두 번째 성공하는 테스트에서 최종 `SUCCESS`와 카카오 전송이 확인되고, 최종 실패 테스트에서 재시도 대기 3회가 확인된다. |

| F-10 | DB 이력 저장 실패를 조용히 성공 처리하지 않는다. | High | `SUCCESS` 저장 실패는 DB 저장만 재시도되고 카카오 전송은 중복되지 않는다. 최종 `FAIL` 저장 실패 시 이후 공지 처리를 중단하고 `last_seen`이 전진하지 않는다. |

## AI 기능 검증

기존 이미지 분석용 `Summarizer.analyze_image()` 호출을 유지하며 프롬프트나 모델 동작은 변경하지 않는다. 따라서 신규 AI 정답 샘플셋은 해당 없음. 회귀 기준은 기존 이미지 분석 호출이 유지되는지 여부다.

## 기술 스택

Python 배치 코드, Playwright 기반 크롤링, 카카오 메시징, 기존 pytest 테스트를 사용한다. 신규 외부 서비스나 인프라 변경은 없다.

## 제약사항

| 항목 | 내용 |
|------|------|
| 호환성 | DB schema, KakaoAuth 동작 변경 없음. last_seen은 `FAIL`도 처리 완료 이력으로 간주해 같은 실패 공지를 다음 배치에서 반복 처리하지 않는다. |
| 보안 | Google 토큰 파일 의존성이 사라져 공지 처리 경로의 시크릿 요구가 감소 |
| 성능 | Google Photos 업로드 HTTP 호출 제거로 이미지 공지 처리 시간이 감소 |
| 운영 | Google Photos 모듈 파일은 남겨두며, 이번 범위에서는 import 의존성만 제거 |

## 공개 전환 시나리오

시크릿 관련 변경이다. 공지 처리 경로가 `config/google_token.json`을 요구하지 않게 되어 공개 전환 시 운영자가 Google 토큰 파일을 준비하거나 노출할 가능성이 줄어든다. 기존 문서나 코드에 남은 앨범 URL/ID 제거는 이번 필수 범위는 아니지만, 후속 정리 후보다.

## 테스트 전략

| 레벨 | 대상 | 방법 |
|------|------|------|
| 단위 | `_process_notice_board()` | 공지 이미지 처리 시 Google Photos 인자 없이 다운로드/분석/카카오/DB 흐름 검증 |
| 단위 | `_process_notice_board()` | 공지 처리 일시 실패 후 재시도 성공 검증 |
| 단위 | `_process_notice_board()` | 공지 처리 최종 실패 시 `FAIL` DB 저장, 처리 커서 전진, 다음 게시글 계속 처리 검증 |
| 단위 | `_process_notice_board()` | 다운로드 결과 0장 실패 처리와 DB 저장 실패 정책 검증 |
| 단위 | `run()` | Google Photos 클라이언트 생성/import 의존성이 없는지 정적/테스트로 확인 |
| 회귀 | batch tests | `batch/tests` 관련 테스트 실행 |

## 사용자 검증 게이트

내부 시범 사용으로 검증한다. 운영자가 `config/google_token.json` 없이 배치를 1회 실행하고, 이미지 포함 공지의 카카오 요약 수신과 DB 기록을 확인한다. 확인 결과는 릴리스 전 체크리스트 또는 산출물 보고에 기록한다.
